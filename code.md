# 개발 기초 (code) 강의계획서

## 1. 교과목 개요

- **과목 소개**: <컴퓨터 기초(basic)>를 이수하여 컴퓨터를 기본적으로 다룰 수 있는 학생이 수강하는 **개발자 진입 과정**입니다. 프로그래밍 언어 문법(Python·Java 등)을 배우기에 앞서, 어떤 분야의 개발자가 될 것인지 방향을 잡고, 개발 환경을 직접 구축하고, 코드를 올바르게 작성하는 컨벤션을 익히며, 협업 도구(Git·GitHub)로 팀 개발에 참여하는 **개발자로서의 기초 체력**을 완성합니다.
- **학습 목표**: 개발의 세계 지형도를 이해하고, 실제 개발 환경을 스스로 셋업하며, 코드를 일관된 규칙(컨벤션)으로 작성하고, Git·GitHub, LLM 활용까지 개발자로 성장하기 위한 필수 기반 역량을 갖춥니다.
- **선수 과목**: 컴퓨터 기초(basic) — 파일 시스템, 경로, OS 기본 제어, 터미널 기초

---

## 2. 학습 성과(LO)

- 개발 분야(웹·앱·AI·서버 등)의 지형도를 설명하고, 본인의 관심 분야와 진입 경로를 제시할 수 있다.
- 개발 환경(IDE·런타임·PATH 환경변수)을 이해하고 언어별 실행 환경을 스스로 셋업할 수 있다.
- 코드 네이밍·들여쓰기·주석·디렉토리 구조 등 개발 컨벤션을 준수하며 코드를 작성할 수 있다.
- LLM 기반 바이브 코딩 워크플로우를 이해하고 AI 생성 코드를 비판적으로 검토할 수 있다.
- Git으로 커밋·브랜치·병합을 수행하고, GitHub에서 Pull Request와 Issue로 팀 협업을 운영할 수 있다.

### 2주차
- **대주제**: 개발 환경 이해 — IDE·에디터·런타임의 실체
- **세부학습목표**: "코드를 실행한다"는 행위가 어떤 소프트웨어 계층을 거치는지 정확히 이해하고, 이번 학기 실습 환경을 스스로 구성한다.

#### 📌 2-1. 개발 도구의 스펙트럼
1. 텍스트 에디터(VSCode·Sublime Text) — 가볍고 확장 가능한 편집 도구
2. 통합 개발 환경(IDE: IntelliJ·PyCharm·Eclipse) — 빌드·디버그·테스트 일체형 환경
3. 클라우드 IDE(GitHub Codespaces·Replit) — 설치 없이 브라우저에서 코딩
4. 언어별 관례적 도구 — Python→VSCode/PyCharm, Java→IntelliJ, iOS→Xcode
5. 에디터 vs IDE 선택 기준 — 학습 초기에는 VSCode, 규모 커지면 언어 전용 IDE

#### 📌 2-2. 컴파일 언어 vs 인터프리터 언어
1. 컴파일 언어(C·Java) — 소스 코드를 기계어로 변환(컴파일) 후 실행하는 2단계 구조
2. 인터프리터 언어(Python·JavaScript) — 소스 코드를 줄 단위로 즉시 해석 실행
3. JVM(Java Virtual Machine) — 바이트코드를 OS 위에서 실행하는 가상 기계
4. 런타임(Runtime) — 코드를 실행하는 물리적 환경(Python 인터프리터·Node.js 등)
5. 같은 Python 코드가 버전에 따라 실행 안 될 수 있는 이유 — 런타임 버전 관리

#### 📌 2-3. Python 실행 환경 첫 셋업
1. Python 3 공식 홈페이지에서 OS별 올바른 설치 파일 선택 방법
2. 설치 시 "Add Python to PATH" 체크박스가 핵심인 이유
3. 터미널에서 `python --version` 또는 `python3 --version` 설치 확인
4. `print("Hello, World!")` 파일(.py) 저장 후 터미널 실행까지 전체 파이프라인
5. REPL(Read-Eval-Print Loop) — 대화형 Python 셸에서 한 줄씩 즉시 실행하는 방법

- **강의 내용**: 에디터·IDE·런타임 개념, 컴파일·인터프리터 차이, Python 설치
- **실습 내용**: Python 설치 → `python --version` 확인 → `print("Hello")` 파일 실행 전체 체험
- **과제**: Python 실행 환경 셋업 인증 + "코드를 실행하기까지 어떤 단계가 있었는가" 단계별 설명 리포트

### 3주차
- **대주제**: 환경 변수(PATH) 완전 정복 — 명령어가 왜 이 실행되지 않는가
- **세부학습목표**: `'python' is not recognized` 에러의 원인을 PATH 구조로 이해하고 직접 해결할 수 있다.

#### 📌 3-1. 환경 변수의 개념과 역할
1. 환경 변수(Environment Variable) — OS가 관리하는 전역 키-값 설정 저장소
2. PATH — 명령어를 찾을 디렉토리 목록(콜론·세미콜론으로 구분)
3. OS가 명령어를 탐색하는 순서 — PATH 배열 앞에서부터 순차 탐색
4. `echo $PATH`(Mac·Linux) / `echo %PATH%`(Windows)로 현재 PATH 출력
5. 환경 변수의 유효 범위 — 현재 터미널 세션만 vs 시스템 전체 영구 등록 차이

#### 📌 3-2. OS별 PATH 등록 방법
1. Windows: 시스템 속성 → 고급 → 환경 변수 → PATH 편집 경로
2. macOS·Linux: `~/.zshrc` 또는 `~/.bashrc` 파일에 `export PATH=$PATH:/새경로` 추가
3. `source ~/.zshrc` — 변경 사항을 현재 터미널에 즉시 반영하는 이유
4. `which python`(Mac·Linux) / `where python`(Windows) — 실제 실행되는 파일 경로 확인
5. Python 2와 Python 3가 공존할 때 `python` vs `python3` 혼동 해결

#### 📌 3-3. PATH 오류 트러블슈팅
1. `command not found` / `'python' is not recognized` — PATH에 등록되지 않은 원인
2. 터미널 재시작 없이 PATH 변경이 반영되지 않는 상황과 `source` 해결법
3. 가상 환경(venv) 활성화 시 PATH 앞에 가상환경 경로가 자동 삽입되는 원리
4. 동일 명령어가 여러 위치에 설치된 경우 어떤 것이 실행되는가 — PATH 우선순위
5. 고의로 PATH를 잘못 설정하여 에러를 재현하고 원인을 분석하는 실습

- **강의 내용**: 환경 변수 개념, PATH 탐색 순서, OS별 등록 방법
- **실습 내용**: PATH에 없는 위치 파일로 `command not found` 고의 발생 → PATH 수정으로 해결
- **과제**: PATH 오류 재현 캡처 + 수복 과정 설명 포함 트러블슈팅 리포트

### 5주차
- **대주제**: 코드 작성 기초와 컨벤션 — 올바르게 쓰는 코드의 기준
- **세부학습목표**: 코드를 단순히 실행되게 쓰는 것을 넘어, 협업하는 동료와 미래의 나 자신이 읽기 좋은 코드를 작성하는 기준과 원칙을 익힌다.

#### 📌 5-1. 네이밍 컨벤션 (Naming Convention)
1. camelCase — 두 번째 단어부터 대문자 시작, JavaScript 변수·함수 표준 (`getUserName`)
2. PascalCase — 모든 단어 대문자 시작, Python·Java 클래스 이름 표준 (`UserProfile`)
3. snake_case — 단어를 언더스코어로 연결, Python 변수·함수 표준 (`get_user_name`)
4. kebab-case — 단어를 하이픈으로 연결, URL·CSS 클래스·파일 이름에 주로 사용
5. SCREAMING_SNAKE_CASE — 모두 대문자+언더스코어, 상수(Constant) 네이밍 표준 (`MAX_RETRY_COUNT`)

#### 📌 5-2. 들여쓰기·공백·코드 구조
1. 들여쓰기(Indentation) — 탭(Tab) vs 스페이스 2·4칸 논쟁과 언어별 표준
2. Python에서 들여쓰기가 문법인 이유 — 블록 구분이 중괄호가 아닌 들여쓰기로 결정
3. 한 줄 최대 글자 수 — PEP8(79자)·Google Style(80자)·Black(88자) 기준 비교
4. 공백 줄의 역할 — 논리적 단위를 시각적으로 분리하는 가독성 도구
5. Magic Number 금지 — `if score >= 60:` 대신 `PASS_SCORE = 60`으로 의미 부여

#### 📌 5-3. 주석·문서화·프로젝트 구조
1. 좋은 주석 vs 나쁜 주석 — `# x에 1을 더함` (나쁨) vs `# 다음 자동 증가 ID 선점` (좋음)
2. 주석은 '무엇'이 아닌 '왜'를 설명해야 하는 이유
3. Docstring — Python 함수·클래스에 사용법을 문서화하는 표준 방식
4. README.md — 프로젝트의 목적·실행 방법·기여 방법을 설명하는 필수 문서
5. 프로젝트 디렉토리 구조 — `src/`·`tests/`·`docs/`·`data/` 역할 분리 컨벤션

- **강의 내용**: 네이밍 컨벤션, 들여쓰기·공백 기준, 주석·문서화·디렉토리 구조
- **실습 내용**: 컨벤션 위반 코드를 제공하여 모두 수정하는 '코드 리팩토링' 미션
- **과제**: 본인이 작성한 간단한 Python 코드를 PEP8 컨벤션에 맞게 전체 재작성하여 비포·애프터 비교 제출

### 6주차
- **대주제**: 터미널 개발 활용 — 파이프라인과 자동화
- **세부학습목표**: 반복적인 작업을 CLI 파이프라인과 쉘 스크립트로 자동화하는 개발자 워크플로우를 체득한다.

#### 📌 6-1. 표준 입출력과 리다이렉션
1. 표준 입출력 — stdin(입력)·stdout(출력)·stderr(에러) 3채널 구조
2. `>` 리다이렉션 — 명령어 출력을 파일에 저장 (`ls > list.txt`)
3. `>>` 추가 리다이렉션 — 기존 파일에 이어 쓰기 (덮어쓰기 방지)
4. `2>` 에러 리다이렉션 — 에러 메시지만 별도 파일에 저장
5. `/dev/null` — 출력을 버리는 블랙홀 경로 활용

#### 📌 6-2. 파이프(|)로 명령어 체이닝
1. 파이프(`|`) — 앞 명령어의 stdout을 뒤 명령어의 stdin으로 직접 연결
2. `grep "ERROR" log.txt` — 특정 패턴이 포함된 줄 필터링
3. `sort` / `uniq -c` — 정렬·중복 제거·개수 집계
4. `wc -l` — 줄 수 세기 / `head -n 10` / `tail -n 10` — 상하단 출력
5. 여러 파이프 조합 원라이너: `cat log.txt | grep "ERROR" | sort | uniq -c | sort -rn`

#### 📌 6-3. 쉘 스크립트 기초
1. shebang(`#!/bin/bash`) — 스크립트 파일의 실행 인터프리터 지정
2. 변수 선언과 사용 (`NAME="hojin"` → `echo $NAME`)
3. `if [ 조건 ]; then ... fi` — 조건 분기 기초 문법
4. `for` 반복문으로 파일 목록 순환 처리하는 패턴
5. `chmod +x 스크립트.sh` 실행 권한 부여 후 `./스크립트.sh` 실행

- **강의 내용**: stdin·stdout·stderr, 파이프·리다이렉션, 쉘 스크립트 기초
- **실습 내용**: 로그 파일에서 ERROR 키워드 추출·정렬·카운팅 원라이너 작성, 자동화 스크립트 제작
- **과제**: 반복하고 싶은 작업을 쉘 스크립트로 구현하여 제출 (예: 날짜별 백업 폴더 자동 생성)

#### 📌 7-1. 패키지 생태계의 구조

#### 📌 7-2. 가상 환경(Virtual Environment)의 필요성

3. 가상환경 활성화 — `source venv/bin/activate`(Mac·Linux) / `venv\Scripts\activate`(Win)
4. `pip freeze > requirements.txt` — 현재 환경 패키지 목록을 파일로 저장
5. `pip install -r requirements.txt` — 다른 컴퓨터에서 동일 환경 재현하는 협업 흐름

#### 📌 7-3. 패키지 활용 실전과 보안
1. `requests` — 공개 API 호출 3줄 코드로 날씨·환율 데이터 받아오기
2. 버전 핀닝(`==2.0.0`) — 버전 명시 없이 설치하면 나중에 코드가 깨지는 이유
3. `pip list` / `pip show 패키지명` — 설치된 패키지 목록·상세 정보 확인
4. `pip-audit` / `npm audit` — 설치된 패키지의 보안 취약점 검사
5. 패키지 삭제 — `pip uninstall`·`npm uninstall`과 `requirements.txt` 동기화 유지

### 9주차

#### 📌 9-1. LLM 기반 코딩 도구의 작동 원리
1. LLM(Large Language Model) — 수천억 개 파라미터로 코드 패턴을 학습한 언어 모델
2. GitHub Copilot·ChatGPT·Cursor·Gemini Code — 기능 비교와 각 도구의 사용 시나리오
3. 컨텍스트 윈도우(Context Window) — AI가 한 번에 참고할 수 있는 텍스트 양의 한계
4. AI 코드 생성이 확률 기반인 이유 — 동일 프롬프트도 결과가 달라지는 원리
5. 환각(Hallucination) — 존재하지 않는 함수·라이브러리를 자신 있게 만들어내는 현상

#### 📌 9-2. 바이브 코딩(Vibe Coding)이란
1. 바이브 코딩 정의 — Andrej Karpathy가 제안한 "AI에게 의도를 말하고 결과를 검토하는" 개발 방식
2. 전통적 코딩 vs 바이브 코딩 비교 — 문법 암기 중심 vs 의도·구조 설계 역량 중심
3. 효과적인 프롬프트 작성법 — 맥락(Context)·제약(Constraints)·예시(Examples)를 명확히
4. 바이브 코딩의 위험 — 이해 없는 코드 복붙이 유지보수에 미치는 장기적 비용
5. "AI를 주니어 개발자처럼 쓰기" — AI가 생성한 코드를 반드시 리드하는 역할 유지

#### 📌 9-3. AI 코딩 도구 실전 패턴
1. 코드 생성 요청 — 함수 단위로 작게 나눠 요청하고 테스트 후 조합하기
2. 코드 설명 요청 — "이 코드가 어떻게 동작하는지 줄별로 설명해줘"
3. 리팩토링 요청 — "이 함수를 PEP8 컨벤션에 맞게 개선하되 동작은 동일하게 유지해줘"
4. 에러 해결 요청 — 에러 전문 + 코드 컨텍스트를 함께 붙여 원인 질문하기
5. AI 응답 검증 — 생성 코드 직접 실행·테스트 + 공식 문서 교차 확인 루틴

- **강의 내용**: LLM 코딩 도구 원리, 바이브 코딩 개념, AI 활용 검증 전략
- **실습 내용**: Cursor 또는 ChatGPT로 간단한 Python 프로그램을 대화형으로 완성하며 AI 코드를 단계적으로 검증·수정하는 워크플로우 체험
- **과제**: AI 생성 코드 1개를 리뷰 — ①올바른 부분 ②수정 필요 부분 ③이해 못 한 부분 분류 코드 리뷰 리포트 제출

3. 최소 재현(Minimal Reproduction) — 최소한의 코드로 버그를 단독 환경에서 재현
4. `try – except` 방어 코드 — 예상 가능한 오류를 미리 처리하는 습관
5. AI 디버깅 활용법 — 스택 트레이스를 붙여넣고 컨텍스트와 함께 원인 물어보기

- **대주제**: Git 버전 관리 1 — 타임머신의 작동 원리
- **세부학습목표**: 코드의 변경 이력을 체계적으로 추적하는 Git의 3단계 공간 구조를 이해하고 커밋 타임라인을 직접 만든다.

#### 📌 11-1. 버전 관리의 필요성
1. 파일 이름으로 버전 관리하는 방식의 5가지 문제점 (최종_진짜최종 패턴)
2. Git이 파일 내용을 스냅샷(Snapshot) 단위로 저장하는 원리
3. 분산 버전 관리(DVCS) — 모든 팀원이 전체 히스토리를 로컬에 보유
4. `git config --global user.name/user.email` 최초 사용자 정보 설정
5. `.git` 폴더 — Git이 모든 히스토리를 관리하는 숨겨진 저장소

#### 📌 11-2. Git의 3공간 구조
1. Working Directory — 실제 작업하는 파일들이 있는 일반 폴더
2. Staging Area(Index) — 다음 커밋에 포함할 파일을 예약하는 임시 공간
3. Local Repository — 커밋이 영구 기록으로 저장되는 타임라인 공간
4. `git add 파일명` — Working Directory → Staging Area 전송
5. `git commit -m "메시지"` — Staging Area → Repository 영구 기록

#### 📌 11-3. 커밋 히스토리 탐색과 활용
1. `git log --oneline` — 커밋 히스토리 한 줄 요약 조회
2. `git diff` — 스테이징 전 변경 내용 비교 / `--staged` 옵션으로 스테이징 후 비교
3. `git status` — Working Directory와 Staging Area 상태 한눈에 확인
4. `.gitignore` 파일 — 추적에서 제외할 파일·폴더 패턴 등록 방법
5. `git restore 파일명` — 수정 내용을 마지막 커밋 상태로 되돌리기

- **강의 내용**: 버전 관리 필요성, Git 3공간 구조, 커밋 타임라인
- **실습 내용**: 텍스트 파일 3번 수정하며 각 단계마다 커밋 → `git log --oneline` 확인
- **과제**: 일주일치 일기를 매일 커밋하여 7개 커밋 로그 히스토리 스크린샷 제출

- **세부학습목표**: 브랜치 생성과 병합 충돌 해결, GitHub를 통한 원격 저장소 동기화와 PR 협업 사이클을 한 주에 체득한다.

#### 📌 12-1. 브랜치와 병합
1. 브랜치 = 타임라인의 평행 분기점 — 실험적 변경을 본진에 영향 없이 진행

3. `git merge 브랜치명` — 다른 브랜치의 커밋을 현재 브랜치에 통합
4. 충돌 마커(`<<<<<<<`, `=======`, `>>>>>>>`) 각각의 의미 해석
5. `git log --oneline --graph --all` — 브랜치 분기·병합 흐름 시각화

#### 📌 12-2. GitHub 원격 저장소 연결
1. `git remote add origin URL` — 로컬 저장소에 원격 저장소 주소 등록
2. `git push origin main` — 로컬 커밋을 원격으로 업로드
3. `git pull origin main` — 원격 최신 커밋을 로컬로 병합
4. `git clone URL` — 원격 저장소 전체를 로컬로 복사
5. SSH 키 인증 설정 — 매번 비밀번호 없이 GitHub 연결하는 방법

#### 📌 12-3. Pull Request와 Issue 기반 협업
1. Fork → 브랜치 작업 → PR 제출 → 코드 리뷰 → Merge 전체 협업 사이클
2. 좋은 커밋 메시지 컨벤션 — `feat:`, `fix:`, `docs:` 유형 접두사(Conventional Commits)
3. Issue 트래커 — 버그 보고·기능 요청을 카드로 관리하는 방법
4. README.md — 마크다운 기본 문법으로 프로젝트 소개 문서 작성
5. `Closes #이슈번호` — PR 본문에 작성하여 Merge 시 Issue 자동 종료 연결

- **강의 내용**: 브랜치·병합·충돌 해결, GitHub push·pull, PR·Issue 협업
- **실습 내용**: 짝 실습 — 상대방 레포 Fork → 오타 수정 PR → 리뷰 댓글 → Merge 교환

2. StackOverflow 답변 날짜 확인 — 3년 이상 된 답변은 라이브러리 버전 확인 필수
3. 공식 문서(Python Docs·MDN)에서 정확한 API 확인하는 방법
4. 에러 메시지가 너무 길 때 핵심 키워드만 추출하는 방법
5. AI로 해결 후 공식 문서로 교차 검증하는 표준 루틴

#### 📌 13-3. 트러블슈팅 리포트 작성법
1. 문제 상황 기술 — 언제·어떤 코드를·어떻게 실행했는가
2. 에러 전문 첨부 — 스크린샷보다 텍스트로 붙여넣기가 검색·공유에 효율적
3. 시도한 해결 방법 나열 — 어떤 접근을 했고 왜 안 됐는가
4. 최종 해결 방법 + 참조 URL 명시
5. 재발 방지 — 같은 에러를 다음에 만나지 않으려면 무엇을 기억해야 하는가

- **강의 내용**: 스택 트레이스 해독, 구글링 전략, 트러블슈팅 리포트 작성
- **실습 내용**: 3가지 에러 유형 코드를 받아 분석 → 구글링 → 수정까지 제한 시간 내 완성
- **과제**: 에러 발생 → 원문 → 검색어 → 참조 URL → 해결 과정 포함 '트러블슈팅 일지' 3건 제출
